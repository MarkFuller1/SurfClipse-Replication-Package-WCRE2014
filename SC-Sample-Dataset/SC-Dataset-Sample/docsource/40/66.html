Description Tim Mok 2008-04-10 10:31:50 EDT 
With the code for authenticated HTTP connections, there has been excessive prompting for login details. I tested with the Ganymede test site (http://update.eclipse.org/testUpdates). It seems that the user is prompted from various mirrors (some include FTP sites) for authentication details. When canceling the user/pass dialog, it seems to find another mirror that doesn't require a user/pass. When attempting to retrieve another file, the same mirror is used and another prompt given to the user despite canceling the dialog a moment ago.
Comment 1 John Arthorne 2008-04-10 14:19:05 EDT 
What should happen is the failure gets reported back to MirrorSelector#reportResult. This will increase the failure count for that mirror, causing it to be never selected again unless every single mirror has also failed.

Comment 3 Tim Mok 2008-04-11 12:01:15 EDT 
Created attachment 95704 [details]
Fix v02

Patch changes:

+Added the getConnectionContext(*) to the download(*) in ECFMetadataTransport to mirror the code in ECFTransport.
+Added a check for incorrect authentication information.

The 401 check works in my testing. However, there isn't an error code returned when incorrect authentication details are given. It results in a "java.net.ProtocolException: Server redirected too many  times (20)". I added in the check for it but I'm still concerned about the HTTP connection being attempted 20 times before returning an exception. It seems possible a server might lock out a connection well before 20 attempts. It seems to be a limitation of java.net attempting the redirect (and failing) and the HTTP protocol that a 401 code isn't required to be returned when authentication information is incorrect.
Comment 4 John Arthorne 2008-04-11 13:58:18 EDT 
CCing Scott for insight on handling of 401. We seem to be getting the raw failure messages back from java.net, and without further context the best we can do is parse the exception message for particular substrings. This is of course brittle, and for example will likely not work if we switch to apache HTTP client. Is there any possibility for the ECF layer to be providing us a richer failure exception with a code, so we can check for a 401 authentication failure without parsing the message itself?
