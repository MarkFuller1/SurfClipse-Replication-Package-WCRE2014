SingleSingOutFilter failure to get logoutRequest POST param results in redirect loops

        Description

        org.jasig.cas.client.session.SingleSignOutFilter.doFilter(ServletRequest, ServletResponse, FilterChain):
        in case of POST: final String logoutRequest = CommonUtils.safeGetParameter(request, "logoutRequest");

        org.jasig.cas.client.util.CommonUtils.safeGetParameter(HttpServletRequest, String):
        request.getQueryString() == null || request.getQueryString().indexOf(parameter) == -1 ? null : request.getParameter(parameter);
        query string does NOT contain "logoutRequest", its a POST param --> single sign out filter is not applied

        results (the same as if not having Single Sign Out Filter installed at all):

          Juergen           added a comment - 31/Aug/09 1:00 PM

          wouldn't it be better to make org.jasig.cas.client.util.CommonUtils.safeGetParameter(HttpServletRequest, String) more robust, able to handle POST requests as well? other filters using this method might also be invoked on POST and not properly handle such a request.

          cf. http://www.ja-sig.org/wiki/display/CASUM/Applications+that+Require+POST+Responses

          on the other hand, it's not good trying to be more clever than the request.getParameter() method. The request could be custom wrapped so request.getParameter() could return a value for a param, not found in either query string or request POST body. So performance related introduction of safeGetParameter might have been a bad idea, simply calling request.getParameter(parameter) should do the job.
