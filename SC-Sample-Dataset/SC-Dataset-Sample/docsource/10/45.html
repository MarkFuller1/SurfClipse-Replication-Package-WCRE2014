in an application without

Submitted by DonH on Sun, 09/30/2012 - 02:53.

in an application without security services enabled, the application will deploy but any java services (including a database service) will fail to be found and the application will not work. If the security service is enabled then the application will not deploy since a security related filter initialization fails,

The errors include things like ...
Error creating bean with name 'authenticationManager'
Error creating bean with name 'jdbcDaoImpl' defined in ServletContext resource [/WEB-INF/project-security.xml]: Cannot resolve reference to bean 'YOUR DATABASE SERVICEDataSource' while setting bean property 'dataSource'
No ServiceWire found for service; check spring.xml file for ReflectServiceWire "runtimeService"

I believe the problem is related to how wavemaker attempts to dynamically load the spring.xml files for services.
in the project-springapp.xml file wavemaker uses the following construct...

<import resource="classpath:**/*.spring.xml"/>

From my understanding the intention is that spring will search the classpath for resources which match the wildcard and load them into the application context.
The wildcard **/*.spring.xml should mean all files ending in .spring.xml in any level of any directory on the classpath. This appears to work fine in Tomcat, but in Glassfish 3.1 it doesn't find the spring.xml files for the wavemaker services such as runtimeService, securityService, and any custom java services you might have created including a database.

I'm still not 100% certain what is causing the difference but it could possibly be related to the classloader implementation of glassfish. My experience with these technologies is a bit light, so it would be best if someone having solid spring/glassfish experience could offer their insight.

However, for the short-term, I have found that changing the construct slightly seems to resolve the problem.

in the WEB-INF/project-springapp.xml
change the one line from

<import resource="classpath:**/*.spring.xml"/>

to

<import resource="classpath*:**/*.spring.xml"/>

Yes, the added "*" goes in front of the colon character.

this change is surprisingly difficult to make. It seems that the project-springapp.xml file in the project source directory is replaced when you do a deploy.
Its a bit frustrating that because the deployment fails, glassfish removes the uncompressed war file from the domain directory so it isn't there already uncompressed and ready for the change.
so take whatever approach works best for your situation. After building the war/ear file tear it back apart to make the change, or change the template file in the studio application directory, or ???
