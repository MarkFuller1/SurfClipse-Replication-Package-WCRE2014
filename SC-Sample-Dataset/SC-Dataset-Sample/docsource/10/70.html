        You shouldn't...

        If you have classes which use log4j hardcoded, instead of slf4j, you need the log4j-over-slf4j to reroute teh logging to slf4j which could then use for instance logback as the logging implementation. If you don't you don't need that jar. (there is also a similair approach for commons-logging to redirect that logging to slf4j).

        In general you want to use slf4j in favor of commons-logging, it behaves better in osgi enabled environments and uses a better mechanism for detecting which logging implementation it has to use. It also behaves better in regards to classloading and memory management.

        I suggest you read the slf4j documentation it explains all you need quite clearly imho...

              Remove commons-logging dependency and include the slf4j wrapper for commons-logging. That way everything will be routed through slf4j and you have a single entry point.

              Commons-logging is, just as slf4j, a logging api it is not a logging implementation.
