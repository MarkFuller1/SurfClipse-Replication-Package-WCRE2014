I was able to duplicate the problem with the Amazon's web service. The root cause is with the version of Axis we use. The latest version of Axis can process the WSDL just fine, but it creates different java code. We'll have to rework the internals of WebORB to make it compatible with the latest version of Axis. It will be available in the next release. A workaround I can suggest for now, is to process your WSDL using wsdl2java from Axis2 and expose the generated Java code as a remoting service through weborb.

Mark

Hi Hans_dc,

Here's a workaround I'd try:

1. Generate Java code from your WSDL using whatever WSDL-2-Java methodology/technology you feel comfortable with
2. Check if the generated code contains a class representing the service. The class should have the same public methods as they are defined in to the WSDL
3. Compile the code and deploy into WebORB. To deploy, you can either copy the class files into WEB-INF/classes (make sure to maintain the directory structure) or create a JAR file and copy it into WEB-INF/lib.
4. Start WebORB and see if you can locate the service class (the one from step 2) in the Services tab of the management console. Check if you can invoke the methods on the class

If all of the above has worked, then you can invoke your webservice by invoking the methods on the class from step 2.

Hope this helps.

The problem appears to be that you have getter/setter methods when you don't have an associated property.
It's expecting bean style classes. Add a property with same name as your getter.


public class MyClass {
// add this property to make the servicebrowser work:
private String dayOfMonth;

public String getDayOfMonth() {
return "Saturday";
}
}

It's a shame it doesn't reflect the other direction (look at properties and then attempt to find accessors)
