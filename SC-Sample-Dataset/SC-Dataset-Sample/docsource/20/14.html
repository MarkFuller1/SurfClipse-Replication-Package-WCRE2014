Chapter 9.  Known Issues with this release

Following is a list of known issues at the time of release.

Included Components

These are the known issues for the components included with this distribution of JBoss Web Framework Kit.
Maven Repository
BZ#745763

The JBoss Enterprise Application Platform Maven repository is a prerequisite to use the JBoss Web Framework Kit Maven repository on JBoss Enterprise Web Server as well as the Enterprise Application Platform.
Hibernate Search
BZ#820191

The Javadoc documentation for ClassBridge should mention Norms instead of StoreNorm as returned type.
RichFaces
BZ#821640

RichFaces cannot be built from source with the JBoss Web Framework Kit 2 zip, due to missing and conflicting dependencies.
BZ#802022

Compiling RichFaces UI exceeds the standard limit of maximum opened files on UNIX based machines. To workaround this issue, raise the ulimit to at least 8192 in /etc/security/limits.conf.
BZ#801048

Compiling RichFaces UI components logs two bad faces-config.xml errors that do not affect functionality of the build.
BZ#769022

Packaging Hibernate 3.3 and RichFaces 3.3 libraries in an application using Enterprise Web Server causes the security check to fail, due to a combination of signed and unsigned jars. This is because the Hibernate jars in Enterprise Web Server come signed, and Hibernate has a transitive dependency on commons-collections. RichFaces has a transitive dependency on commons-collections, and commons-digester, but transitive dependencies are not shipped. The workaround for this issue is to use the community version of commons-collections within an application that also uses RichFaces.
BZ#760115

Using Spring WebFlow 2.3 and RichFaces 4 together requires an extra bean to resolve RichFace resources, but this bean is not automatically part of faces:resources and gives a 404 error when a CSS file is requested. The workaround for this issue is to change the order of flowMapping to 2, and add the following configuration: 
<bean name="richfacesResourceHandler"
class="org.springframework.faces.webflow.JsfResourceRequestHandler" />

    <bean
class="org.springframework.web.servlet.handler.SimpleUrlHandlerMapping">
        <property name="order" value="1" />
        <property name="mappings">
            <value>
                /rfRes/**=richfacesResourceHandler
            </value>
        </property>
    </bean>
Snowdrop
BZ#820740

Snowdrop source cannot be build out of the box because it requires artifacts which are not present in Maven Central nor distributed Enterprise Maven repositories.

To workaround this issue, enable the JBoss Nexus repository by including the following snippet in the settings.xml file: 
<profile>
      <id>jboss-public-repository</id>
      <repositories>
        <repository>
          <id>jboss-public-repository-group</id>
          <name>JBoss Public Maven Repository Group</name>
          <url>https://repository.jboss.org/nexus/content/groups/public-jboss/</url>
          <layout>default</layout>
          <releases>
            <enabled>true</enabled>
            <updatePolicy>never</updatePolicy>
          </releases>
          <snapshots>
            <enabled>true</enabled>
            <updatePolicy>never</updatePolicy>
          </snapshots>
        </repository>
      </repositories>
      <pluginRepositories>
        <pluginRepository>
          <id>jboss-public-repository-group</id>
          <name>JBoss Public Maven Repository Group</name>
          <url>https://repository.jboss.org/nexus/content/groups/public-jboss/</url>
          <layout>default</layout>
          <releases>
            <enabled>true</enabled>
            <updatePolicy>never</updatePolicy>
          </releases>
          <snapshots>
            <enabled>true</enabled>
            <updatePolicy>never</updatePolicy>
          </snapshots>
        </pluginRepository>
      </pluginRepositories>
    </profile>


  <activeProfiles>
    <activeProfile>jboss-public-repository</activeProfile>
  </activeProfiles>

More information on setting the public repository is available here.
Examples
BZ#834009

Importing an Errai-based example into JBoss Developer Studio will not work out of the box. Follow this procedure to make it deployable from an IDE:
Procedure 9.1. Errai example

  1. Install Google Plugin and Web Toolkit

    Go to JBoss Central, select Software/Updates tab and install required software.

  2. Add Google Web Toolkit Entry Module

    Select project, right click on Properties, go to Google/Web Toolkit tab and add Entry point module. Dialog will offer you existing modules, for example "HelloWorld".

  3. Compile Google Web Toolking classes

    Select project, right click on the Google/GWT Compile.

  4. Add Errai Marshalling Output to the Deployment Assembly

    Select project, right click on Properties, go to Deployment Assembly tab and add following: 
                         /target/classes  WEB-INF/classes

  5. Add Google Web Toolkit Compiler Output to the Deployment Assembly

    Select project, right click on Properties, go to Deployment Assembly tab and add following: 
                         /target/${finalName}  /

    ${finalName} depends on your project settings. For example, "errai-bus-demos-helloworld-2.0.1.Final-redhat-1

    Result
    The application can be deployed in an IDE.
BZ#832096

The following are deviations and extra information on the README file:

  • Chapter Optional Components is not required for the quickstarts to run.

  • Follow the instructions in the Maven Repository User Guide for setting up the settings.xml file.

  • Helloworld Google Web Toolkit example README in section Application in Google Web Toolkit Developer Mode step 2, should mention helloworld-gwt instead of helloworld-errai.
BZ#832087

The RichFaces Validation example should use 4.2.0.Final-redhat-1 version instead of 4.2.0.Final for the hibernate-validator artifact. This does not impair functionality of the example.
BZ#832066

JBoss Developer Studio throws errors about JavaScript files distributed with the example being invalid. Functionality is not impaired, but the error can be removed by excluding specific JS files (5.cache.js) from being validated.
BZ#831606

The TicketMonster application does not contain an Administration section, which means that a 404 Error is shown when the Administration menu is clicked.
BZ#822210

It is not possible to return from details view to list view in Kitchensink on mobile devices.
BZ#822205

After a new member is created in the application using a browser on an iPhone device, the list of new members will fail to update correctly, even when creating a new member from a desktop web browser.
BZ#802503

The JAR file sportsclub-domain.jar cannot be resolved to persistence.xml in sportsclub-jpa-dao/src/main/resources/META-INF. This does not influence the application itself.
BZ#802444

Both the default installation of JBoss Enterprise Application Platform and JMS Push, which is required by RichFaces Showcase example, are secured by default. To workaround this issue and to deploy the application, add a user to a secured ApplicationRealm:

  1. Add a new user ApplicationRealm with the $JBOSS_HOME/bin/add-user.bat/sh script, with the role: guest

  2. Modify web.xml in the RichFaces Showcase. Replace $<username> and $<password> with valid credentials: 
    <context-param>
        <param-name>org.richfaces.push.jms.connectionUsername</param-name>        <param-value>${username}</param-value>
</context-param>
<context-param>
       <param-name>org.richfaces.push.jms.connectionPassword</param-name>
       <param-value>${password}</param-value>
</context-param>
BZ#801406

During the import of the Petclinic project in JBoss Developer Studio, a default persistence.xml file with invalid contents is created under src/main/resources/META-INF. During deployment, this will supersede the correct one, to be found in src/main/environments/eap/resources/META-INF and will cause the deployment to fail. The workaround for this issue is to remove the src/main/resources/META-INF/persistence.xml file created during import.
BZ#787751

The Petclinic application uses JSTL expressions which may fail to evaluate in environments that use newer Unified EL implementations. The expressions known to be in violation are: ${visit.new} found in src/main/webapp/WEB-INF/jsp/pets/visitForm.jsp and ${pet.new} found in src/main/webapp/WEB-INF/jsp/pets/form.jsp.

If encountering an error message of the form: 
${<variable>.new}" contains invalid expression(s): javax.el.ELException: Failed to parse the expression [${<variable>.new}],

any occurrence of ${<variable>.new} needs to be replaced with ${<variable>['new']}. For details, please consult the src/main/webapp/WEB-INF/jsp/owners/form.jsp page of the same application.

Certified Components

The following are the known issues for the components certified for use with JBoss Web Framework Kit.
Spring
BZ#807578

Packaging Hibernate 3.3 and Spring Roo libraries in an application causes the security check to fail, due to a combination of signed and unsigned jars. This is because the Hibernate jars in Enterprise Web Server come signed, and Hibernate has a transitive dependency on commons-collections.

Spring Roo uses Struts Tiles, an artifact that has a transitive dependency on commons-beanutils, which is unsigned and contains same classes as commons-collections.

The workaround for this issue is to use the community or unsigned version of commons-collections within an application that also uses Spring Roo.
BZ#801304

Securing an application in the following way creates a security bug by transitively depending on a compromised OpenIDJava: 
<dependency>
  <groupId>org.springframework.security</groupId>
  <artifactId>spring-security-openid</artifactId>
  <version>2.0.7.RELEASE</version>
</dependency>

To workaround this issue, use the following configuration alongside the original: 
<dependency>
  <groupId>org.openid4java</groupId>
  <artifactId>openid4java</artifactId>
  <version>0.9.6</version>
</dependency>
BZ#796313

The SpringContextResourceAdaptor present in Spring 2.5.6.SEC03, Spring 3.0.7.RELEASE, and Spring 3.1.1.RELEASE violates section 5.3.1 of the JCA 1.5 specification, and does not fully implement JCA 1.5 and 1.6 contact.

The workaround for this problem is so disable archive validation within the JCA subsystem. Use this configuration to do so: 
                  
        <subsystem xmlns="urn:jboss:domain:jca:1.1">
            <archive-validation enabled="false" />
        </subsystem>
BZ#785684

When using Spring Bill of Materials (BOM) distributed in JBoss Web Framework Kit together with JBoss Enterprise Web Server, users have to add following dependency to act as a coordinator for Web Services: 
               <dependency>
<groupId>com.sun.xml.wsit</groupId>
    <artifactId>wsit-rt</artifactId>
    <version>1.3.1</version>
</dependency>

This artifact is available in JBoss Nexus Repository. Please follow instructions in bug #831607 in order to enable the repository.
BZ#749170

The location of the MBean server has changed between versions of JBoss Enterprise Application Platform, so using the following configuration fails with deployment errors: 
<bean id="mBeanServer" class="org.jboss.mx.util.MBeanServerLocator"
factory-method="locateJBoss"/>

The workaround for this issue is to use this configuration instead: 
<bean id="mbeanServer"
class="org.springframework.jmx.support.MBeanServerFactoryBean">
      <property name="locateExistingServerIfPossible" value="true" />
</bean>
Struts 1
BZ#803860

Struts 1 applications without the struts.xml descriptor deploys succesfully on JBoss Enterprise Application Platform 6, however Actions are not properly mapped. There is no known workaround for this issue other than including struts.xml file.
BZ#803749

The JSF integration in the Struts 1 Showcase application does not work with JBoss Enterprise Application Platform 6. The workaround for this issue is to override default JSF implementation.

Do this by adding following configuration fragment into web.xml configuration file of your Struts 1 Showcase example: 
<context-param>
   <param-name>org.jboss.jbossfaces.WAR_BUNDLES_JSF_IMPL</param-name>
   <param-value>true</param-value>
</context-param>
Google Web Toolkit
BZ#803782

iPad, iPhone 4, and Google Nexus S fail to activate and write text into RichTextArea element. The devices treated the text area as uneditable space. There are no reported problems with the element's toolbar, buttons, or select boxes.
BZ#803780

iPad and iPhone devices ignore single-tap events outside of popped-up elements. Popups do not disappear when the user navigates away. The following workarounds are available for listed GWT elements:
Image

Single tap directly on the image
Date Picker

Pick a date.
Simple Text Area

Activate different elements on the page
Other

Navigate to a different page and back

Confirmed Frameworks

These are the known issues for the confirmed frameworks, which will run on JBoss Enterprise products but are not supported.
Grails
BZ-790873

OSGi is not enabled in JBoss Enterprise Application Server 6. Artifacts produced by Grails miss a Package-Import of javax.naming in the bundle. The workaround for this issue is to disable OSGi while packaging your Grails application. You can do this by setting the following property in the BuildConfig.groovy configuration file:

grails.project.war.osgi.headers = false
Struts 2
BZ#803747

The JSF integration in the Struts 2 Showcase application does not work with JBoss Enterprise Application Platform 6. The workaround for this issue is to override default JSF implementation.

Do this by adding following configuration fragment into web.xml configuration file of your Struts 1 Showcase example: 
<context-param>
   <param-name>org.jboss.jbossfaces.WAR_BUNDLES_JSF_IMPL</param-name>
   <param-value>true</param-value>
</context-param>

Technical Previews

TorqueBox
BZ#831724

TorqueBox requires installation of JRuby 1.6.7.2 which does not work on Solaris. This prevents TorqueBox Ruby gems being installed correctly, so TorqueBox support for Solaris is not included in this release of JBoss Web Framework Kit.
BZ#833464

Attempting to deploy a Ruby application to TorqueBox in the standalone-full-ha.xml with the polyglot-hasingleton extension and subsystem enabled results in the following error and rollback of the deployment: 
               09:33:47,449 INFO  [org.jboss.as.server] (DeploymentScanner-threads - 2) JBAS015870: Deploy of deployment "blah-knob.yml" was rolled back with failure message {"JBAS014771: Services with missing/unavailable dependencies" => ["jboss.deployment.unit.\"blah-knob.yml\".ha-singleton.coordinator Missing[jboss.jgroups.channel.hasingleton]"]}

The workaround is to remove or comment out the following two elements from standalone-full-ha.xml: 
<extension module="org.projectodd.polyglot.hasingleton"/>
<subsystem xmlns="urn:jboss:domain:polyglot-hasingleton:1.0"/>

As a result TorqueBox will not support HASingleton jobs and services and all jobs and services defined in a torquebox.yml need to include the singleton: false parameter or they will not run when clustered
BZ#831820

When compiling TorqueBox from sources using JDK 7, the maven-compiler-plugin version should be updated in torquebox-parent pom from 2.0.2 to 2.5.1.
