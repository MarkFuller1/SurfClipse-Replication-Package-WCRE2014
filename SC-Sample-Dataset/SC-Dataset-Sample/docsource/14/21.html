        From looking at the source of PageContextImpl.initApplicationContext,
        the issue is acquiring a session-scoped lock (line 2551:
        https://github.com/getrailo/railo/blob/3.3.1.000/railo-java/railo-core/src/railo/runtime/PageContextImpl.java#L2551)
        before initializing the Application scope and then the Session scope
        (though both are gated to only run if needed). The call to
        PageContextImpl.initApplicationContext in ModernAppListener._onRequest
        (line 101: https://github.com/getrailo/railo/blob/3.3.1.000/railo-java/railo-core/src/railo/runtime/listener/ModernAppListener.java#L101)
        is not gated, so it fires (and thence attempts to acquire the locks)
        on every request.

        As near as I can tell, these locks should be VERY short-lived except
        when the application and/or session actually needs initialization.
        Our access logs indicate there is literally zero concurrency on the
        box, often with many minutes (if not hours) between requests (it's a QA box). Certain
        user/request combinations (mostly file uploads) always create this
        exception, while the same request from different machines are fine,
        and different requests from the same machine are also fine.

        We thought we'd managed to reproduce it in dev and addressed the issue we manifested by
        removing onApplicationStart from our Application.cfc (replacing it
        with homegrown initialization/locking code). While that did fix
        the issue we'd manifested, the InterruptedExceptions have continued, so we clearly didn't fix the right thing.
