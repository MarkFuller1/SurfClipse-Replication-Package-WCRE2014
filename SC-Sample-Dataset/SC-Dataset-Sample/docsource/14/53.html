Luc Boudreau added a comment - 25/Oct/12 10:03 AM - edited
As a side note to this case, this happens when a statement is canceled and we try to send out a "ExecutionEnd" event to the Mondrian monitor. To fix this, we should do something like this in Execution.checkCancelOrTimeout:

boolean needInterrupt = false;
try {
  if (Thread.interrupted()) {
    needInterrupt = true;
    fireExecutionEndEvent();
  }
} finally {
  if (needInterrupt) {
    Thread.currentThread().interrupt();
  }
  throw MondrianResource.instance().QueryCanceled.ex();
}

Benny Chow added a comment - 25/Oct/12 12:31 PM
Luc, Analyzer will not interrupt a thread directly but it will call query.cancel(). When this is called, Mondrian should interrupt the current thread and Thread.interrupted() should return true. If it does not return true, then it is possible that the prior report's exceptions (including InterruptedException) show up in the current report's query log.

Can you check if there is a possibility that a query cancel throws an InterruptedException without actual calling Thread.currentThread().interrupt()? If this is a valid scenario, I can improve Analyzer's exception checking to ignore InterruptedException. This way if a report is cancelled while a new one running, the just cancelled report's InterruptedException will not show up in the new report.

I have seen this issue too but only once in a blue moon.
