        • MSSCCI Provider 1.2 and the MSSCCPRJ.SCC file. Todd, MSSCCPRJ.SCC file is not related to specific SCC provider, it is part of SCC API used to hook up Visual Studio to Source Code Control provider conforming to SCC API specs. SCC provider (in your case TFS SCC provider) creates that file locally and stores there SCC-specific context information - mapping of your project local folder to TFS server, server path etc. In older versions of SCC (in Visual Studio 6.0 for example), that information was stored directly in project files which created a whole bunch of problems; in order to solve those issues SCC API starting from v 1.1 introduced SCC file that is usually created (on per project basis) when you add your code to SCC repository or perform "Open from SC". Maintaining that file is the responsibility of SCC provider, so "leave at alone" is the best you can do :) Hope that helps. Re: Phoenix Fatal Error in Instrumenting Managed Code

Andy Ayers - MSFT

When you compile -clr: oldSyntax, you end up getting a fair amount of native code linked into the binary. In order for the Phoenix to properly handle this code, make sure you link with the -profile option, as shown below:

Code Snippet
cl -clr:oldSyntax -Zi test.cpp -link -profile

The assert about the RVA only seems to appear when the test program is compiled with Phoenix. I think it can safely be ignored. I don't see the other asserts you show for some reason.

If you then try to run test-mtrace.cpp, you may get a dialog popup about being unable to locate msvcm80.dll. This happens because mtrace does not properly handle the case where the module being instrumented has an external manifest -- this manifest must be copied and renamed. You can do this manually, however:

Code Snippet
copy test.exe.manifest test-mtrace.exe.manifest

and then running test-mtrace.exe works as expected:
